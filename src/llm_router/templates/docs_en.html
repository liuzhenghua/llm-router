API Integration

Request Headers

All endpoints share the same request header specification. Authentication uses Authorization: Bearer.

Header Required Description
Authorization Required Format: Bearer <api-key>. Use the key created in the API Keys page.
x-request-id Optional Custom request ID for end-to-end tracing. If not provided, the gateway auto-generates one and returns it in the response header.
x-end-user Optional End-user identifier for request log attribution. Pass a user ID or session ID for auditing.
x-channel Optional Request channel for log classification. Falls back to the API Key's configured default channel. Examples: PERSONAL, TEAM, BOT.
anthropic-version Optional Anthropic protocol only. Defaults to 2023-06-01, passed through to upstream.
anthropic-beta Optional Anthropic protocol only. Enables Beta features; passed through to upstream when provided.

OpenAI — Chat CompletionsOfficial Docs ↗

llm-router is fully compatible with the OpenAI Chat Completions protocol. Point base_url to this service to use any OpenAI-compatible client or SDK.

Endpoint: POST {{ base_url }}/v1/chat/completions

Example Code

from openai import OpenAI

client = OpenAI(
  base_url="{{ base_url }}/v1",
  api_key="sk-xxxxxxxxxxxxxxxx",
)

response = client.chat.completions.create(
  model="your-logical-model",
  messages=[{"role": "user", "content": "Hello!"}],
)
print(response.choices[0].message.content)
curl {{ base_url }}/v1/chat/completions \
  -H "Authorization: Bearer sk-xxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{"model":"your-logical-model","messages":[{"role":"user","content":"Hello!"}]}'
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "{{ base_url }}/v1",
  apiKey: "sk-xxxxxxxxxxxxxxxx",
});

const response = await client.chat.completions.create({
  model: "your-logical-model",
  messages: [{ role: "user", content: "Hello!" }],
});
console.log(response.choices[0].message.content);

OpenAI — List / Get ModelsOfficial Docs ↗

List logical models accessible to the current API Key, or retrieve a single model's details. Response format is fully compatible with OpenAI's /v1/models.

Endpoints:

  • GET {{ base_url }}/v1/models — list all available models
  • GET {{ base_url }}/v1/models/{model_id} — retrieve a single model

Example Code

# List all models
curl {{ base_url }}/v1/models \
  -H "Authorization: Bearer sk-xxxxxxxxxxxxxxxx"

# Retrieve a single model
curl {{ base_url }}/v1/models/your-logical-model \
  -H "Authorization: Bearer sk-xxxxxxxxxxxxxxxx"
from openai import OpenAI

client = OpenAI(
  base_url="{{ base_url }}/v1",
  api_key="sk-xxxxxxxxxxxxxxxx",
)

# List all models
models = client.models.list()
for m in models.data:
  print(m.id)

# Retrieve a single model
model = client.models.retrieve("your-logical-model")
print(model.id, model.created)
Response Format: List endpoint returns {"object":"list","data":[{"id":"...","object":"model","created":1234567890,"owned_by":"llm-router"}]}. Detail endpoint returns a single model object with the same fields.

OpenAI — EmbeddingsOfficial Docs ↗

llm-router supports the OpenAI Embeddings protocol to convert text to vector representations. Only routes to Providers configured with an OpenAI Endpoint.

Endpoint: POST {{ base_url }}/v1/embeddings

Example Code

from openai import OpenAI

client = OpenAI(
  base_url="{{ base_url }}/v1",
  api_key="sk-xxxxxxxxxxxxxxxx",
)

response = client.embeddings.create(
  model="your-embedding-model",
  input="Hello, world!",
)
print(response.data[0].embedding[:5])
curl {{ base_url }}/v1/embeddings \
  -H "Authorization: Bearer sk-xxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{"model":"your-embedding-model","input":"Hello, world!","encoding_format":"float"}'
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "{{ base_url }}/v1",
  apiKey: "sk-xxxxxxxxxxxxxxxx",
});

const response = await client.embeddings.create({
  model: "your-embedding-model",
  input: "Hello, world!",
});
console.log(response.data[0].embedding.slice(0, 5));
Note: Embedding requests only route to Providers with openai_endpoint configured (Anthropic Endpoint is not supported). Billing is based on input tokens only (no output tokens). Response format is fully compatible with OpenAI /v1/embeddings.

Anthropic — MessagesOfficial Docs ↗

llm-router also supports the Anthropic Messages protocol. Point base_url to this service's /anthropic prefix.

Endpoint: POST {{ base_url }}/anthropic/v1/messages

Example Code

import anthropic

client = anthropic.Anthropic(
  base_url="{{ base_url }}/anthropic",
  api_key="sk-xxxxxxxxxxxxxxxx",
)

message = client.messages.create(
  model="your-logical-model",
  max_tokens=1024,
  messages=[{"role": "user", "content": "Hello!"}],
)
print(message.content[0].text)
curl {{ base_url }}/anthropic/v1/messages \
  -H "x-api-key: sk-xxxxxxxxxxxxxxxx" \
  -H "anthropic-version: 2023-06-01" \
  -H "Content-Type: application/json" \
  -d '{"model":"your-logical-model","max_tokens":1024,"messages":[{"role":"user","content":"Hello!"}]}'
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic({
  baseURL: "{{ base_url }}/anthropic",
  apiKey: "sk-xxxxxxxxxxxxxxxx",
});

const message = await client.messages.create({
  model: "your-logical-model",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Hello!" }],
});
console.log(message.content[0].text);

Developer Tools

Claude CodeOfficial Docs ↗

Proxy Claude Code API requests through llm-router via environment variables, allowing you to use any backend model. Set ANTHROPIC_BASE_URL to this service's /anthropic prefix.

Environment Variables

Shell
# Replace kimi-k2.6 with your desired logical model name
export ANTHROPIC_BASE_URL={{ base_url }}/anthropic
export ANTHROPIC_AUTH_TOKEN=${YOUR_API_KEY}
export ANTHROPIC_MODEL=kimi-k2.6
export ANTHROPIC_DEFAULT_OPUS_MODEL=kimi-k2.6
export ANTHROPIC_DEFAULT_SONNET_MODEL=kimi-k2.6
export ANTHROPIC_DEFAULT_HAIKU_MODEL=kimi-k2.6
export CLAUDE_CODE_SUBAGENT_MODEL=kimi-k2.6
export ENABLE_TOOL_SEARCH=false
# Start Claude Code
claude
Tip: Replace kimi-k2.6 with the logical model name created in Logical Models; replace ${YOUR_API_KEY} with the key created in API Keys.

Operations Guide

Quick Start

Follow these steps to complete the initial setup and start routing LLM requests through llm-router.

1

Create a Provider

Go to the Providers page and click New Provider. Enter the provider name, downstream API Key, model name, endpoint URL, and token pricing. Once saved, this Provider acts as the underlying model node that handles requests.

2

Create a Logical Model and Bind Routes

Go to Logical Models and click New Logical Model. Enter the model name exposed to clients (the value clients pass as the model parameter).
After creation, click the Routes button on the logical model row. In the side panel, add one or more route rules linking this logical model to your Providers.

3

Create an API Key and Authorize Models

Go to API Keys and click New API Key. Enter a name, initial balance, and QPS limit.
In the Allowed Logical Models field, select or type the models this key may access. After saving, the full key is displayed once — save it securely.

After completing these three steps, you can use the generated API Key to send requests to llm-router via OpenAI or Anthropic protocols. All requests will be forwarded to the appropriate Provider based on routing rules.

Routing Rules

llm-router supports three routing strategies, which can be combined under the same logical model.

⚡

Priority Routing

Assign a priority value to each route (lower = higher priority). The router tries routes in priority order, falling through to the next only when a higher-priority node is unavailable. Ideal for primary/failover scenarios: e.g. primary at priority=1, backup at priority=2.

⚖️

Weighted Routing

Routes sharing the same priority are distributed by weighted random selection based on their weight values. Ideal for traffic splitting: e.g. two Providers each at weight=50 each handle 50% of traffic.

🛡️

Fallback Routing

When a route's is_fallback flag is set, that node is only selected when all non-fallback nodes are unavailable. Ideal for last-resort failsafe: e.g. a low-cost model as the final fallback to ensure service continuity.

Route Selection Order: Group by priority ascending → within each group, weighted random selection → if all nodes in a group fail, proceed to the next group → after all non-fallback groups fail, try fallback nodes. Degraded routes (e.g. repeated timeouts) are temporarily removed from the candidate pool and automatically restored after a cooldown period.
