API 接入文档

请求头

所有接口均使用相同的请求头规范，认证方式统一为 Authorization: Bearer。

Header 必填 说明
Authorization 必填 格式为 Bearer <api-key>，使用在 API Keys 页面创建的密钥。
x-request-id 可选 自定义请求 ID，用于全链路追踪。若不传，网关会自动生成并在响应头中返回。
x-end-user 可选 终端用户标识，用于请求日志归因。可传入用户 ID 或会话 ID 便于后续审计。
x-channel 可选 请求渠道标识，用于日志分类与归因。若不传则回落到该 API Key 配置的默认渠道。常用值示例：PERSONAL、TEAM、BOT。
anthropic-version 可选 仅 Anthropic 协议有效，默认 2023-06-01，会透传至上游。
anthropic-beta 可选 仅 Anthropic 协议有效，用于开启 Beta 功能，有值时透传至上游。

OpenAI — Chat Completions

llm-router 完全兼容 OpenAI Chat Completions 协议。将 base_url 指向本服务即可使用任何支持 OpenAI 格式的客户端或 SDK。

Endpoint: POST {{ base_url }}/v1/chat/completions

示例代码

from openai import OpenAI

client = OpenAI(
  base_url="{{ base_url }}/v1",
  api_key="sk-xxxxxxxxxxxxxxxx",
)

response = client.chat.completions.create(
  model="your-logical-model",
  messages=[{"role": "user", "content": "Hello!"}],
)
print(response.choices[0].message.content)
curl {{ base_url }}/v1/chat/completions \
  -H "Authorization: Bearer sk-xxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{"model":"your-logical-model","messages":[{"role":"user","content":"Hello!"}]}'
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "{{ base_url }}/v1",
  apiKey: "sk-xxxxxxxxxxxxxxxx",
});

const response = await client.chat.completions.create({
  model: "your-logical-model",
  messages: [{ role: "user", content: "Hello!" }],
});
console.log(response.choices[0].message.content);

OpenAI — List / Get Models

获取当前 API Key 可访问的逻辑模型列表，或查询单个模型详情。响应格式与 OpenAI /v1/models 完全兼容。

Endpoints:

  • GET {{ base_url }}/v1/models — 列出所有可用模型
  • GET {{ base_url }}/v1/models/{model_id} — 查询单个模型

示例代码

# 列出所有模型
curl {{ base_url }}/v1/models \
  -H "Authorization: Bearer sk-xxxxxxxxxxxxxxxx"

# 查询单个模型
curl {{ base_url }}/v1/models/your-logical-model \
  -H "Authorization: Bearer sk-xxxxxxxxxxxxxxxx"
from openai import OpenAI

client = OpenAI(
  base_url="{{ base_url }}/v1",
  api_key="sk-xxxxxxxxxxxxxxxx",
)

# 列出所有模型
models = client.models.list()
for m in models.data:
  print(m.id)

# 查询单个模型
model = client.models.retrieve("your-logical-model")
print(model.id, model.created)
响应格式： 列表接口返回 {"object":"list","data":[{"id":"...","object":"model","created":1234567890,"owned_by":"llm-router"}]}。 详情接口返回单个模型对象，字段相同。

OpenAI — Embeddings

llm-router 兼容 OpenAI Embeddings 协议，可将文本转换为向量表示。仅路由到配置了 OpenAI Endpoint 的 Provider。

Endpoint: POST {{ base_url }}/v1/embeddings

示例代码

from openai import OpenAI

client = OpenAI(
  base_url="{{ base_url }}/v1",
  api_key="sk-xxxxxxxxxxxxxxxx",
)

response = client.embeddings.create(
  model="your-embedding-model",
  input="Hello, world!",
)
print(response.data[0].embedding[:5])
curl {{ base_url }}/v1/embeddings \
  -H "Authorization: Bearer sk-xxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{"model":"your-embedding-model","input":"Hello, world!","encoding_format":"float"}'
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "{{ base_url }}/v1",
  apiKey: "sk-xxxxxxxxxxxxxxxx",
});

const response = await client.embeddings.create({
  model: "your-embedding-model",
  input: "Hello, world!",
});
console.log(response.data[0].embedding.slice(0, 5));
说明： Embedding 请求仅路由到配置了 openai_endpoint 的 Provider（不支持 Anthropic Endpoint），billing 按输入 Token 计费（无输出 Token）。 响应格式与 OpenAI /v1/embeddings 完全兼容。

Anthropic — Messages

llm-router 同样兼容 Anthropic Messages 协议。将 base_url 指向本服务的 /anthropic 前缀即可。

Endpoint: POST {{ base_url }}/anthropic/v1/messages

示例代码

import anthropic

client = anthropic.Anthropic(
  base_url="{{ base_url }}/anthropic",
  api_key="sk-xxxxxxxxxxxxxxxx",
)

message = client.messages.create(
  model="your-logical-model",
  max_tokens=1024,
  messages=[{"role": "user", "content": "Hello!"}],
)
print(message.content[0].text)
curl {{ base_url }}/anthropic/v1/messages \
  -H "x-api-key: sk-xxxxxxxxxxxxxxxx" \
  -H "anthropic-version: 2023-06-01" \
  -H "Content-Type: application/json" \
  -d '{"model":"your-logical-model","max_tokens":1024,"messages":[{"role":"user","content":"Hello!"}]}'
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic({
  baseURL: "{{ base_url }}/anthropic",
  apiKey: "sk-xxxxxxxxxxxxxxxx",
});

const message = await client.messages.create({
  model: "your-logical-model",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Hello!" }],
});
console.log(message.content[0].text);

编程工具集成

Claude Code

通过环境变量将 Claude Code 的 API 请求代理到 llm-router，从而使用任意后端模型。 ANTHROPIC_BASE_URL 需指向本服务的 /anthropic 前缀。

环境变量配置

Shell
# 将kimi-k2.6替换成你想用的模型
export ANTHROPIC_BASE_URL={{ base_url }}/anthropic
export ANTHROPIC_AUTH_TOKEN=${YOUR_API_KEY}
export ANTHROPIC_MODEL=kimi-k2.6
export ANTHROPIC_DEFAULT_OPUS_MODEL=kimi-k2.6
export ANTHROPIC_DEFAULT_SONNET_MODEL=kimi-k2.6
export ANTHROPIC_DEFAULT_HAIKU_MODEL=kimi-k2.6
export CLAUDE_CODE_SUBAGENT_MODEL=kimi-k2.6
export ENABLE_TOOL_SEARCH=false
# 启动 Claude Code
claude
提示：将 ${YOUR_LOGIC_MODEL} 替换为在 Logical Models 页面创建的逻辑模型名称；将 ${YOUR_MOONSHOT_API_KEY} 替换为在 API Keys 页面创建的密钥。

运维文档

操作指引

按照以下步骤完成首次配置，即可开始通过 llm-router 路由 LLM 请求。

1

创建 Provider（模型供应商）

前往 Providers 页面，点击 新增 Provider。填写供应商名称、下游 API Key、模型名称及 Endpoint 地址，并配置 Token 单价。保存后该 Provider 将作为实际承接请求的底层模型节点。

2

创建逻辑模型并绑定路由

前往 Logical Models 页面，点击 新增逻辑模型，填写对外暴露的模型名称（即客户端 model 参数的值）。
创建完成后，点击该逻辑模型行右侧的 路由 按钮，在弹出面板中添加一条或多条路由规则，将逻辑模型与已创建的 Provider 关联起来。

3

创建 API Key 并授权模型

前往 API Keys 页面，点击 新增 API Key。填写名称、初始余额及 QPS 限制。
在 允许使用的模型 字段中，勾选或输入允许该 Key 访问的逻辑模型名称。保存后将显示完整密钥（仅展示一次），请妥善保存。

完成以上三步后，即可使用生成的 API Key 通过 OpenAI 或 Anthropic 协议向 llm-router 发起请求，所有请求将根据路由规则被转发到对应的 Provider。

路由规则

llm-router 支持三种路由策略，可在同一逻辑模型下组合使用。

⚡

优先级路由

为每条路由设置 priority 数值（越小越优先）。Router 会按优先级从低到高依次尝试，只有当高优先级节点不可用时才会 fallthrough 到下一个。 适合 主备切换 场景：例如主节点 priority=1，备用节点 priority=2。

⚖️

权重路由

相同 priority 的多条路由之间，按 weight 比例进行加权随机分配。 适合 流量分发 场景：例如两个 Provider 各 weight=50 则各承载 50% 流量。

🛡️

后备路由（Fallback）

勾选路由的 is_fallback 标志后，该节点仅在所有非 fallback 节点均不可用时才会被选中。 适合 兜底保障 场景：例如低成本模型作为兜底，确保服务不中断。

路由选择顺序： 按 priority 升序分组 → 同组内按 weight 加权随机 → 组内全部失败则进入下一优先级组 → 所有非 fallback 组失败后尝试 fallback 节点。 当路由节点处于降级状态时（如连续超时），会被暂时从候选池中移除，并在一段时间后自动恢复。
